面试官问：你这个项目最大的难点是什么？

你可以这样说：

这个项目最大的难点其实不在 OCR 或 Elasticsearch 本身，而在于整个图片处理链路是异步、跨服务、且不可控的。

我们一开始是用串行调用来做的，但很快发现只要中间任一步失败，整个流程就会卡死，而且无法恢复。

所以后来我把“图片”当成一种资源，而不是一次请求，设计了一套多阶段状态流，用 Redis 记录过程状态，用 MySQL 保证最终结果可持久化。

这样即使 OCR 回调延迟、Embedding 失败或者 ES 写入异常，都可以基于状态进行重试或补偿，保证索引最终一致。

这个设计的代价是状态管理复杂度上升，但换来的是链路可观测性和可恢复性，这是我认为在工程上更重要的。

注意看，这段话：
没讲“我当时怎么 debug”
没讲“我怎么配端口”
但工程判断非常清晰
四、关于 KNN 和混合检索，你该怎么“不显得单薄”

你现在简历里那一句：

利用 Elasticsearch 的 KNN 算法…

太像“我知道这个词”。

你应该准备一段面试用解释版本：

我们没有把向量检索当成唯一答案，而是作为语义召回手段。

实际上在文物图片场景里，很多查询仍然依赖明确的关键词，比如年代、作者或主题，所以我们采用的是向量相似度 + 关键词匹配的混合打分。

这样可以避免纯 KNN 在噪声文本或 OCR 不稳定时召回结果失真，同时也保证在模糊语义查询下仍有不错的体验。

一句话总结：

你知道 KNN 的边界，而不是盲目迷信它。

这很加分。
