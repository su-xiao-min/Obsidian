# 🔥 一、JWT + Redis —— 你现在属于“能用，但没想透”

---

## ❌ 1. 双 Token 的理解太浅

你说：

> 单 token 不能主动失效

对，但这只是**表层原因**。

真正的问题是：

|方案|状态存储|可控性|
|---|---|---|
|单 JWT|无状态|不可撤销|
|Session|有状态|可控|
|双 Token|半状态|可控 + 轻量|

双 Token的本质：

> 用短生命周期 Access Token 保证性能  
> 用有状态 Refresh Token 保证控制权

不是“为了无感刷新”，而是：

### 👉 把认证拆成

- **频繁校验部分（无状态）**
- **可撤销部分（有状态）**

---

## ❌ 2. Refresh Token 被盗 —— 你回答太随缘

你说：

> TLS保护

这只是传输层保护。真正问题是：

### 攻击场景：

用户手机被 root  
浏览器 XSS  
数据库泄漏

你要回答的是：

### 正确思路：

- Refresh Token 必须：
    - HttpOnly
    - SameSite
    - 绑定设备指纹
    - 存 hash，而不是明文
- 支持 Refresh Token 轮换（rotation）

每刷新一次：

旧 token 作废  
生成新 token

这叫 **Refresh Token Rotation**

否则：

> 攻击者可以无限刷新

## ❌ 3. Redis 宕机 —— 你低估了影响

你说：

> 重新登录就好

这是学生级答案。

企业级答案要说：

- Redis 主从
- 哨兵
- AOF 持久化
- 降级策略

真正问题是：

> 如果 Redis 宕机，Refresh Token 是否全部失效？

如果是，那就是全站强制登出。  
这不是“影响不大”。这是事故。🚨

---

## ❌ 4. 单点登录回答太模糊

你说：

> 加个字段

这不是设计。

真正思路：

### 单点登录实现

- 每个用户只允许一个 Refresh Token
- 新登录时：
    - 删除旧 Refresh Token
    - 写入新的
- Access Token 自然失效

---

## ❌ 5. 强制踢人下线

不是“Redis 能写很多内容”。

而是：

- 维护 token blacklist
- 或删除 refresh token
- 或维护登录版本号（token version）

比如：

JWT 里放 version  
Redis 存当前 version

如果不一致 → 无效

这才是完整回答。

#  二、MD5 秒传 —— 你现在在危险区

你这里最容易被打爆。

---

## ❌ 1. MD5 冲突回答不严谨

你说：

> 有生之年

错误。

MD5 是可以构造碰撞的。  
不是理论，是现实。

正确回答应该是：

- 普通用户无意碰撞概率极低
- 但安全敏感场景不应依赖 MD5
- 可改为 SHA256 或 文件大小 + hash 双重校验

---

## ❌ 2. 前端算 MD5 会不会被篡改？

你说：

> 本地计算怎么篡改

这说明你还没有安全意识。

客户端永远不可信。

攻击者可以：

- 修改 JS
- 直接发请求
- 构造任意 MD5

你必须在后端：

- 再计算一次 hash
- 或对上传完成后的文件校验

否则：

攻击者可以伪造 MD5 直接骗过逻辑上传。

---

## ❌ 3. 引用计数并发安全

你说：

> Redis 单线程

这是非常危险的理解。

问题在于：

- 你的 count 在数据库
- Redis 回写是异步
- 两个请求同时查 MD5 不存在

就会：

两个线程同时创建 originFile

你现在没有任何并发控制设计。

正确答案应包括：

- 唯一索引 (MD5 + bucket)
- 插入失败重试
- 或 乐观锁 / CAS

---

## ❌ 4. count = 0 竞态问题

场景：

线程 A 减到 0  
线程 B 同时新增引用

A 定时任务删除  
B 的引用就悬空

你现在没考虑这个。

正确做法：

- 删除前再次检查 count
- 或使用状态字段
- 或使用事务锁
