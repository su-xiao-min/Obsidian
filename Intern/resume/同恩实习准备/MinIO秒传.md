#  二、MD5 秒传 —— 你现在在危险区

你这里最容易被打爆。

---

## ❌ 1. MD5 冲突回答不严谨

你说：

> 有生之年

错误。

MD5 是可以构造碰撞的。  
不是理论，是现实。

正确回答应该是：

- 普通用户无意碰撞概率极低
- 但安全敏感场景不应依赖 MD5
- 可改为 SHA256 或 文件大小 + hash 双重校验

---

## ❌ 2. 前端算 MD5 会不会被篡改？

你说：

> 本地计算怎么篡改

这说明你还没有安全意识。

客户端永远不可信。

攻击者可以：

- 修改 JS
- 直接发请求
- 构造任意 MD5

你必须在后端：

- 再计算一次 hash
- 或对上传完成后的文件校验

否则：

攻击者可以伪造 MD5 直接骗过逻辑上传。

---

## ❌ 3. 引用计数并发安全

你说：

> Redis 单线程

这是非常危险的理解。

问题在于：

- 你的 count 在数据库
- Redis 回写是异步
- 两个请求同时查 MD5 不存在

就会：

两个线程同时创建 originFile

你现在没有任何并发控制设计。

正确答案应包括：

- 唯一索引 (MD5 + bucket)
- 插入失败重试
- 或 乐观锁 / CAS

---

## ❌ 4. count = 0 竞态问题

场景：

线程 A 减到 0  
线程 B 同时新增引用

A 定时任务删除  
B 的引用就悬空

你现在没考虑这个。

正确做法：

- 删除前再次检查 count
- 或使用状态字段
- 或使用事务锁
