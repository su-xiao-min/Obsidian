### 1. MD5 会冲突吗？
> **你的回答**：会，但不是加密，不怕故意碰撞。正常文件碰撞是有生之年，比系统Bug概率低，不考虑。

*   **深度解析**：
    *   **事实层面**：你说对了一半。MD5 确实存在碰撞，且已被证明可以**人为构造**碰撞（不再是理论上的“有生之年”）。虽然随机两个文件自然碰撞的概率极低（$1/2^{128}$），但如果是**恶意攻击者**，构造两个内容不同但 MD5 相同的文件是非常容易的（已有成熟工具如 `HashClash`）。
    *   **业务风险**：在“秒传”场景下，如果攻击者构造了一个恶意文件 A（病毒/木马）和一个正常文件 B（如 `logo.png`），它们的 MD5 相同。
        1. 你先上传了正常的 B，系统记录 MD5 -> 文件路径。
        2. 攻击者上传恶意的 A，系统检测到 MD5 已存在，直接返回“上传成功”，并让攻击者的记录指向 B 的文件路径。
        3. **结果**：攻击者并没有覆盖文件，但他**成功欺骗了系统**，让他以为上传了 A，实际上数据库里关联的是 B。或者反过来，如果攻击者先上传了伪装成正常文件的恶意文件，后续所有上传该“正常文件”的用户，下载到的都是病毒。
    *   **面试官视角**：你在赌概率，而工程架构讲究**防御性设计**。对于核心数据完整性校验，MD5 已不再安全。
    *   **优化话术**：
        > “理论上 MD5 存在碰撞风险，且可被人为构造。在非高安全敏感场景（如内部 ERP 文档），考虑到性能开销，MD5 是性价比之选。但如果涉及资金、合同等高敏感文件，我会建议升级为 **SHA-256** 或在上传后由服务端进行二次校验（异步计算 SHA-256 比对），以平衡性能与安全。”

### 2. 前端算 MD5 会不会被篡改？
> **你的回答**：前端不懂，本地计算怎么被篡改？

*   **深度解析**：
    *   **核心误区**：**前端代码运行在用户的浏览器里，用户拥有最高控制权**。
    *   **攻击方式**：
        1. **Hook 劫持**：攻击者可以通过浏览器的 Console 或编写插件，Hook 住计算 MD5 的函数，强行返回一个指定的 MD5 值（例如已知存在的某个大文件的 MD5）。
        2. **请求伪造**：攻击者根本不需要走前端的 JS 逻辑，直接使用 Postman 或 Python 脚本构造请求，发送任意文件的流，但在表单参数里填上他想“秒传”的那个文件的 MD5。
    *   **后果**：如果后端完全信任前端传来的 MD5 并直接建立映射，攻击者可以将一个 1KB 的文本文件，通过伪造 MD5，瞬间“秒传”成一个 1GB 的视频文件（实际上数据库指向了那个 1GB 文件的存储地址）。当其他用户下载时，下载到的是 1KB 的文本，导致文件损坏；或者利用此机制探测服务器上是否存在某些敏感文件（盲测）。
    *   **正确做法**：**前端计算的 MD5 仅用于优化体验（预检查），不能作为唯一信任凭证。** 真正的去重逻辑必须依赖**服务端计算**的哈希值，或者在文件落盘后由服务端再次校验。
    *   **优化话术**：
        > “前端计算的 MD5 极易被篡改（通过抓包重放或 Hook 技术），因此**绝不能作为唯一的去重依据**。我的设计中，前端 MD5 仅用于‘预检’接口，告诉用户‘可能’秒传。真正写入数据库建立映射前，MinIO 存储完成后，服务端会（异步或同步）重新计算文件指纹进行二次确认，确保存储内容与指纹一致。”

### 3. 为什么不用 SHA256？
> **你的回答**：麻烦，复杂。

*   **深度解析**：
    *   **技术事实**：在现代 CPU 上，SHA-256 和 MD5 的性能差距非常小（通常相差不到 20%-30%），对于文件 IO 瓶颈来说，这点计算损耗几乎可以忽略不计。说“麻烦、复杂”显得非常不专业，仿佛多写几行代码就是巨大的架构负担。
    *   **真实考量**：通常选用 MD5 是因为历史遗留、兼容旧数据或特定硬件加速支持，而非因为 SHA-256 难用。
    *   **优化话术**：
        > “选择 MD5 主要是出于**历史兼容性**和**生态习惯**（很多旧系统指纹都是 MD5）。实际上 SHA-256 的计算开销在现代服务器上完全可以接受。如果项目重启或对新文件入库，我会优先采用 SHA-256 甚至 BLAKE3，以获得更强的抗碰撞性和安全性，毕竟‘麻烦’不是架构选型的核心理由。”

### 4. 如何防止用户上传恶意覆盖文件？
> **你的回答**：用户水平高能弄出碰撞？而且上传一样 MD5 也不会覆盖，除非预判……很难理解。

*   **深度解析**：
    *   **逻辑漏洞**：你混淆了“哈希碰撞攻击”和“逻辑覆盖攻击”。
        *   **场景一（非碰撞）**：用户 A 上传了 `report.pdf` (MD5: X)。用户 B 想搞破坏，他不需要构造碰撞。他只需要知道 `report.pdf` 的 MD5 是 X，然后上传一个名为 `virus.exe` 但内容也是 `report.pdf` 的文件（或者直接用脚本传）。如果系统设计不当（比如允许不同文件名共用一个物理文件且权限控制不严），可能会导致逻辑混乱。
        *   **场景二（哈希碰撞 - 重点）**：正如第 1 点所述，攻击者构造 `A(正常)` 和 `B(恶意)` 具有相同 MD5。
            1. 攻击者先上传 `B(恶意)`，系统存储 B，记录 MD5->Path(B)。
            2. 正常用户上传 `A(正常)`，系统发现 MD5 已存在，直接复用 Path(B)。
            3. **结果**：正常用户以为自己上传了 A，实际存储的是 B。这就是**恶意覆盖/污染**。
    *   **MinIO 特性**：MinIO 本身是对象存储，Key 通常是 `bucket/object_name`。如果你的去重逻辑是“不同文件名，相同内容，指向同一个 Object Key”，那么一旦 Object 被恶意内容占据，所有引用该 Hash 的文件都坏了。
    *   **正确防御**：
        1. **不可变存储**：MinIO 开启 Object Lock (WORM)，一旦写入不可修改。
        2. **内容寻址 + 命名空间隔离**：不要简单地用 MD5 做文件名。可以使用 `bucket/user_id/hash` 的结构，或者在数据库层做严格的权限校验。
        3. **首次上传可信原则**：对于关键业务，第一次上传该 Hash 的文件需要更严格的审核，或者禁止普通用户触发“新 Hash 的创建”，只能引用已有的。
    *   **优化话术**：
        > “防止恶意覆盖的核心不在于防碰撞，而在于**权限隔离**和**存储策略**。
        > 1. **MinIO 策略**：我开启了 MinIO 的 Object Lock 功能，文件一旦写入不可被修改或覆盖，只能新增版本或删除。
        > 2. **逻辑隔离**：数据库中，文件元数据（文件名、归属用户）与物理路径（Hash）是分离的。即使用户构造了碰撞，他也只能让自己的记录指向一个已存在的物理文件，而无法修改该物理文件的内容（因为 MinIO 不可变）。
        > 3. **审计**：对于首次出现的 Hash，我们会记录详细的上传日志和上传者信息，便于追溯。”

### 5. 引用计数如何保证并发安全？
> **你的回答**：ERP 场景并发不高；Redis 异步回写；Redis 单线程；非分布式。

*   **深度解析**：
    *   **致命缺陷**：**“异步回写” + “引用计数” = 数据不一致的高发区**。
        *   **场景**：
            1. 用户 A 上传文件（Hash: X），DB count=1。
            2. 用户 B 删除文件（Hash: X），DB count 变为 0，触发删除逻辑（或标记删除）。
            3. **就在这一瞬间**，用户 C 上传文件（Hash: X）。
            4. 如果你的逻辑是：删除操作把 count 减到 0 就立刻（或很快）清理物理文件，而上传操作还在异步更新 Redis 或 DB。
            5. **结果**：用户 C 的上传成功了，数据库 count=1，但底层的物理文件可能被删除任务删掉了！或者用户 C 的引用计数更新丢失。
    *   **Redis 单线程≠业务逻辑原子性**：Redis 单线程只保证单个命令原子。如果你需要 `Check Count -> Decrement -> Check if 0 -> Delete` 这一系列动作，必须使用 **Lua 脚本** 才能保证原子性。简单的 `incr/decr` 无法解决“减到 0 时是否需要立即删除物理文件”的竞态问题。
    *   **“并发不高”是借口**：即使是低并发，只要时序错乱，Bug 必现。而且 ERP 中批量导入、定时任务都可能产生瞬时并发。
    *   **正确做法**：
        1. **数据库乐观锁**：`UPDATE file_table SET count = count - 1 WHERE hash = ? AND count > 0`。
        2. **延迟删除（关键）**：`count = 0` 时，**绝对不要立即删除物理文件**。而是更新一个 `delete_at` 时间字段（如当前时间 + 24小时）。
        3. **上传时的逻辑**：上传时发现文件存在，先检查 `delete_at` 字段。如果处于“待删除”状态，将其取消（置空），并 `count + 1`。
        4. **定时任务**：只删除 `delete_at < now()` 且 `count == 0` 的文件。
    *   **优化话术**：
        > “并发安全不能靠‘并发低’来保证。我的方案核心是**‘延迟删除’**机制来消除竞态：
        > 1. 删除操作只是将 `count` 减 1，如果为 0，仅标记 `expire_time`（例如 1 小时后），**不立即物理删除**。
        > 2. 上传操作若命中相同 MD5，发现该文件处于 `expire_time` 状态，则**撤销删除标记**，并将 `count` 加 1。
        > 3. 只有定时任务扫描到 `expire_time` 已过且 `count` 仍为 0 的文件，才执行物理删除。
        > 这样无论上传和删除如何并发交错，都不会出现‘文件被删但计数还在’或‘计数归零但文件被误删’的情况。Redis 仅作为缓存加速计数读取，持久化以 DB 事务为准。”

### 6. count = 0 时如何删除文件？是否会出现竞态？
> **你的回答**：逻辑删除，count=0 什么都不做，写删除时间，30天后定时删。不理解竞态。

*   **深度解析**：
    *   **自我矛盾**：你在第 5 点说“Redis 异步回写”，在第 6 点说“写删除时间”。如果 `count` 的更新是异步的，那么在 `count` 真正变成 0 之前的窗口期，如果有新的上传进来，可能会因为读取到过期的缓存（认为 count!=0）而跳过复用逻辑，导致产生冗余文件；或者反之。
    *   **关于竞态的理解**：你说不理解竞态，这在高级开发面试中是大忌。竞态（Race Condition）是指多个线程/进程访问共享资源，最终结果依赖于执行顺序。
        *   **典型竞态**：线程 A 读到 count=1 -> 线程 B 读到 count=1 -> 线程 A 执行 count-1=0 (触发删除) -> 线程 B 执行 count-1=0 (再次触发删除，报错或异常)。
        *   **或者**：线程 A (删除) 把 count 设为 0 并计划删除 -> 线程 B (上传) 还没读到 count=0，也准备新建文件 -> 导致同一份物理文件被重复上传存储，浪费空间。
    *   **你的方案评价**：你提出的“标记删除时间 + 定时清理”思路是**非常正确**的（这也是业界标准做法，叫 Soft Delete / Grace Period）。这确实能很大程度上缓解竞态。
    *   **需要补充的点**：必须强调**数据库事务**的重要性。`count` 的增减和 `expire_time` 的更新必须在同一个数据库事务中完成，或者使用 Lua 脚本操作 Redis，才能保证中间状态不可见。
    *   **优化话术**：
        > “我采用的正是**软删除 + 延迟清理**策略来解决竞态：
        > 1. 当 `count` 减为 0 时，不物理删除，而是更新 `deleted_at` 字段为当前时间。
        > 2. 此时若有新上传命中该 MD5，SQL 逻辑会检测到 `deleted_at` 不为空，执行 `UPDATE ... SET count = count + 1, deleted_at = NULL`，从而‘复活’文件。
        > 3. 定时任务仅清理 `deleted_at < (Now - 30天)` 且 `count = 0` 的记录。
        > **关键点**：所有的计数加减和状态变更都在**数据库事务**（或 Redis Lua 脚本）中原子执行，确保了即使在极高并发下，‘复活’和‘清理’的逻辑也不会冲突。这就是解决竞态的标准范式。”

### 总结与建议

你的原始回答给人一种"**野路子**"的感觉：
*   **优点**：直觉上知道要用“延迟删除”来处理垃圾回收，知道 MinIO 适合存文件。
*   **缺点**：
    1.  **安全意识薄弱**：低估了前端篡改和哈希碰撞的风险。
    2.  **并发模型模糊**：用“并发低”掩盖架构缺陷，对“异步回写”带来的数据不一致风险缺乏认知。
    3.  **术语使用不当**：说“不理解竞态”是面试大忌。

**修正后的核心逻辑链（建议背诵/理解）：**

1.  **哈希选择**：默认 MD5 兼顾性能，高敏业务可配 SHA-256；知晓碰撞风险但通过**不可变存储**和**首次上传审计**来兜底。
2.  **信任边界**：前端 MD5 仅作**预检提示**，服务端落地后必须**二次校验**或依赖 MinIO 的 ETag 机制确保一致性。
3.  **并发与一致性**：
    *   拒绝“异步回写计数”，采用**数据库乐观锁**或**Redis Lua 脚本**保证计数原子性。
    *   核心大招：**延迟删除（TTL）**。Count=0 仅标记过期时间，给新上传留出“复活”窗口，彻底消除删除与新上传之间的竞态条件。
4.  **防覆盖**：利用 MinIO 的 **Object Lock** 防止物理文件被篡改，利用数据库的**逻辑隔离**防止元数据混乱。
