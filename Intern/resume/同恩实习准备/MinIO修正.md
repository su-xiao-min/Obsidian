### 1. 关于 SHA-256 与恶意覆盖

**你的观点**：用了 SHA-256 就很难构造碰撞，所以很难预测并事先构建碰撞，问题就解决了。
**解析**：

- **基本正确，但有前提**：是的，目前 SHA-256 在工程上被认为是**抗碰撞**的。攻击者无法像 MD5 那样轻松构造出两个不同内容但哈希相同的文件。因此，“预判别人要上传什么文件并提前上传一个伪造的碰撞文件”这种攻击手段，在 SHA-256 下几乎不可行。
- **遗留风险**：虽然不能“构造碰撞”，但如果攻击者**猜到了**某个重要文件的哈希值（例如公开的开源软件安装包），他依然可以上传一个同名或不同名的文件，利用秒传机制“窃取”该文件的存储引用。但这通常属于**信息泄露**范畴，而非**内容篡改**。
- **结论**：升级到 SHA-256 确实能极大缓解“恶意碰撞覆盖”的风险，是推荐做法。但它不能解决**权限控制**带来的逻辑污染问题（即你提到的公共文件被污染）。

### 2. 关于后端校验与预签名策略

**你的困惑**：采用预签名（Presigned URL）直传 MinIO，后端拿不到文件流，怎么做二次校验？
**解析**：
这是一个经典的架构权衡：**性能 vs 安全**。

- **方案 A（完全信任前端）**：前端算 Hash -> 后端查库 -> 有则秒传，无则发预签名 -> 前端直传。
  - _风险_：前端 Hash 可伪造。
- **方案 B（后端二次校验 - 推荐）**：
  1.  前端计算 Hash 发起“预检”。
  2.  若不存在，后端生成预签名 URL，**同时要求前端在上传完成后回调通知后端**（或者利用 MinIO 的 Event Notification / Bucket Notification 功能）。
  3.  **关键步骤**：文件落入 MinIO 后，后端通过 MinIO SDK 读取文件头或部分数据（MinIO 支持流式读取，不需要全部加载到内存），重新计算 SHA-256。
  4.  **比对**：如果重算的 Hash 与前端声称的一致，才正式写入数据库建立映射；如果不一致，直接删除 MinIO 中的文件并报错。
  - _难点_：确实增加了 IO 开销和延迟。
  - _优化_：对于大文件，全量重算太慢。可以只校验文件大小 + 前 1MB 哈希 + 后 1MB 哈希，或者依赖 MinIO 自带的 ETag（通常是 MD5，但可配置）做初步校验。
  - **折中方案**：如果是内部可信网络，且主要防误操作而非防黑客，可以接受前端 Hash。如果是公网 SaaS 服务，必须做服务端校验（至少抽样）。

### 3. 关于权限系统与 `originFile` 表

**你的设计**：`originFile` 只存物理路径，权限由关联表控制。
**解析**：

- **非常正确**。这是标准的**逻辑与物理分离**设计。
- **场景复盘**：
  - 表 `files` (id, minio_path, hash, count, is_public)
  - 表 `user_file_relations` (user_id, file_id, permission_level)
  - 即使黑客通过某种手段（比如利用了未修复的 MD5 碰撞，或者利用了业务逻辑漏洞）让他的记录指向了同一个 `file_id`，只要他在 `user_file_relations` 里没有权限，他就无法下载或查看。
  - **真正的风险点**：如果系统设计允许用户**替换**文件内容（例如“更新版本”功能），而底层实现是直接覆盖 MinIO 对象（Key 不变），那么所有引用该 Key 的用户都会看到被篡改的内容。
  - **对策**：这就是为什么需要 **MinIO 不可变存储** 或 **版本控制**。

### 4. 细说 MinIO 不可变存储 (Object Lock / WORM)

**概念**：WORM (Write Once, Read Many)。一旦文件写入，在设定的保留期内，**任何人都无法修改或删除**，包括 Root 账号和管理员。
**如何实现**：

1.  **开启 Bucket 版本控制**：Object Lock 依赖版本控制。
2.  **设置 Retention Rule**：
    - **Governance Mode（治理模式）**：拥有特殊权限的用户可以解除锁定（适合合规审计）。
    - **Compliance Mode（合规模式）**：锁定期内，**绝对不可删除或修改**，连 AWS/MinIO 管理员也不行（适合法律强监管）。
3.  **效果**：
    - 用户上传文件 A (Hash: X)。MinIO 存储为 `object_v1`。
    - 攻击者试图上传恶意文件 B (Hash: X) 并覆盖。
    - MinIO 会拒绝覆盖 `object_v1`，而是创建一个新版本 `object_v2`（如果开启了版本控制），或者直接报错（如果配置了禁止覆盖）。
    - 你的数据库依然指向 `object_v1`，所以用户下载到的永远是第一次上传的正确文件。
    - **代价**：存储空间会增加（因为保留了历史版本），需要定期清理旧版本（通过 Lifecycle Policy）。

### 5. 命名空间隔离与去重的矛盾

**你的疑问**：如果用了命名空间隔离（如 `user_id/hash`），那多人上传同一文件岂不是存多份？去重失效了？
**解析**：

- **是的，物理上去重失效了**。
- **权衡**：
  - **方案 A（全局去重）**：所有用户共用一个物理文件。
    - _优点_：极度节省空间。
    - _缺点_：权限控制复杂，存在“一人删，全员挂”或“恶意覆盖”风险。
  - **方案 B（逻辑去重 + 物理隔离）**：每个用户存一份物理文件（路径不同），但在数据库层面记录“这个文件是重复的”，前端展示时提示“秒传”，实际上后台还是走了上传流程（或者由后端快速拷贝一份）。
    - _优点_：权限天然隔离，安全。
    - _缺点_：浪费空间。
  - **方案 C（混合模式 - 推荐）**：
    - **私有文件**：物理隔离，不追求极致去重，保安全。
    - **公共文件/系统文件**：全局去重，但必须由**管理员**上传，且开启 Object Lock。普通用户只能“引用”，不能“上传/覆盖”公共库的文件。
    - 这正是你提到的“首次上传可信原则”的落地方式。

### 7. 深度解析：Redis Lua 脚本如何解决竞态？

**你的疑问**：不太理解 Lua 怎么解决，是不是在 Redis 层面避免竞态？

**原理**：
Redis 的单线程模型保证了**单个命令**的原子性。但是，业务逻辑往往需要多个步骤：

1.  `count = get(hash)`
2.  `if count > 0: count = count - 1`
3.  `if count == 0: set_expire_time(hash)`

如果在客户端分三步走：

- 线程 A 执行 step 1 (读到 1)。
- 线程 B 执行 step 1 (读到 1)。
- 线程 A 执行 step 2 (变为 0)，准备执行 step 3。
- 线程 B 执行 step 2 (变为 -1，错误！)，或者线程 B 以为还有文件，结果 A 把文件删了。

**Lua 脚本的作用**：
Lua 脚本在 Redis 中执行时，是**作为一个整体原子执行**的。在执行脚本期间，不会有其他命令插入。

**示例脚本 (`decr_and_check.lua`)**：

```lua
-- KEYS[1]: 文件的 Hash 键
-- ARGV[1]: 过期时间戳

local current_count = tonumber(redis.call('GET', KEYS[1]) or "0")

if current_count <= 0 then
    return -1 -- 异常：计数已经为0或负数
end

local new_count = current_count - 1
redis.call('SET', KEYS[1], new_count)

if new_count == 0 then
    -- 标记删除时间，而不是立即删除物理文件
    -- 这里我们可以设置一个专门的 key 来存删除时间，或者存在 value 里
    redis.call('SET', KEYS[1] .. ":delete_at", ARGV[1])
end

return new_count
```

**调用方式**：
客户端一次性发送这个脚本给 Redis。Redis 会锁住整个执行过程，直到脚本跑完。

- 线程 A 和 线程 B 同时调用。
- Redis 排队执行：先完整跑完 A 的脚本（Count 1->0，标记删除时间），再跑 B 的脚本（此时读到的是 0，返回错误或触发“复活”逻辑）。
- **结论**：是的，Lua 脚本在 **Redis 层面** 彻底消除了多线程并发读写同一个 Key 的竞态条件。

### 8. 关于 Count=0 的理解与清理策略

**你的理解**：

> 把 count == 0 视作文件存在但准备删除（24小时后），删除后再把 count 记录删掉？

**修正与建议**：

- **Count=0 的含义**：表示**当前没有任何活跃的业务记录引用该物理文件**。
- **状态流转**：
  1.  **正常状态**：`count > 0`, `delete_at = null`。
  2.  **待删除状态**：`count = 0`, `delete_at = 2026-03-04 08:00:00` (当前时间 + 24h)。
      - 此时物理文件**依然存在**。
      - 如果有新上传命中该 Hash：检测到 `delete_at` 不为空 -> `count = 1`, `delete_at = null` (**复活**)。
  3.  **物理删除**：定时任务扫描到 `count = 0` 且 `delete_at < now()`。
      - 调用 MinIO 删除物理对象。
      - **删除数据库记录**：是的，此时可以将这条 `files` 表的记录彻底 `DELETE`，或者标记为 `status = deleted` 留作审计。
- **为什么要删记录？**
  - 如果物理文件都没了，留着 `count=0` 的记录没有意义，只会让表越来越大。
  - **注意**：如果为了审计，建议不要物理 `DELETE` 数据库行，而是更新 `status = 'PURGED'`，并保留 `hash` 和 `original_path` 等信息，方便日后查证“某年某月某日有个文件被彻底清除了”。

### 9. 模拟场景：为什么会出现“多删”的情况？（乐观锁的作用）

**场景假设**：没有使用乐观锁，也没有 Lua 脚本，仅仅是简单的 `GET -> DEC -> IF 0 THEN DELETE` 逻辑，且由两个并发线程操作。

**初始状态**：

- 文件 Hash: `X`
- DB Count: `1`
- 物理文件：存在

**并发操作**：

- **线程 A** (用户甲删除文件)
- **线程 B** (用户乙删除文件，假设两人都有权限删，或者这是一个共享文件的并发删除请求)

**时序推演（无锁情况）**：

1.  **T1**: 线程 A 读取 Count = `1`。
2.  **T2**: 线程 B 读取 Count = `1` (此时 A 还没写回)。
3.  **T3**: 线程 A 计算 `1 - 1 = 0`。判断 `0 == 0`，触发**删除物理文件**逻辑。MinIO 文件被删。
4.  **T4**: 线程 A 更新 DB Count = `0`。
5.  **T5**: 线程 B 计算 `1 - 1 = 0` (基于它 T2 读到的 1)。判断 `0 == 0`，触发**删除物理文件**逻辑。
6.  **T6**: 线程 B 尝试删除 MinIO 文件。**报错！** (文件不存在)。
7.  **T7**: 线程 B 更新 DB Count = `0`。

**后果**：

- **报错异常**：线程 B 抛出“文件不存在”异常，可能导致事务回滚，或者日志里一堆错误报警。
- **逻辑混乱**：如果代码没处理好异常，可能会认为“既然报错了，是不是我计数算错了？”从而导致数据不一致。
- **更严重的场景（删除与上传并发）**：
  - 线程 A (删除): 读到 1 -> 减为 0 -> **准备删文件**。
  - 线程 B (上传/复用): 读到 0 (A 已经写回了) -> 认为是新文件 -> **上传新文件** (或者复活逻辑)。
  - 如果 A 的删除动作稍微慢一点（网络 IO），在 B 上传完成后，A 执行了删除。**B 刚上传好的文件被 A 删了！** 这就是典型的竞态导致的数据丢失。

**乐观锁如何解决**：
SQL: `UPDATE files SET count = count - 1 WHERE hash = 'X' AND count > 0`

- **T1**: A 执行 update，条件 `count > 0` 满足，影响行数 1。Count 变为 0。
- **T2**: B 执行 update，此时 DB 中 count 已经是 0。条件 `count > 0` **不满足**。影响行数 0。
- **逻辑判断**：
  - A 发现影响行数 1 -> 继续执行后续逻辑（检查是否需标记删除）。
  - B 发现影响行数 0 -> 知道并发冲突了。重新查询当前状态。
    - 如果发现 count=0 且有 `delete_at`，说明正在删除中，执行“复活”逻辑。
    - 如果发现 count>0，重试减法。
- **结果**：绝对不会出现“两次都去删文件”或者“刚上传就被删”的情况，因为数据库的行锁和条件检查强行串行化了修改计数的操作。

### 总结

你的思考方向非常棒，特别是在权限分离和延迟删除上的直觉很准。

- **SHA-256** 是必须的升级。
- **预签名** 需要配合 **MinIO 事件通知** 或 **上传后回调** 来做服务端校验（至少抽检）。
- **MinIO Object Lock** 是防止物理覆盖的神器。
- **Redis Lua** 或 **DB 乐观锁** 是解决并发计数竞态的标准答案。
- **延迟删除 (TTL)** 是解决“删与增”竞态的核心设计。

把这些点串起来，就是一个高可用、高安全的企业级秒传方案。
