# 一、DataLeap 是什么东西（核心理解）

简单翻译一下 JD：

> **Dataleap = 字节内部的数据开发平台**

它解决的不是：

- 写业务接口
- 写推荐算法

而是：

> **让数据工程师能更容易生产数据。**

换句话说：

```
业务数据库 → 数据采集 → 数据计算 → 数据仓库 → 数据分析
```

DataLeap做的是：

```
把这整条链条做成平台
```

一个更直白的类比：

| 领域     | 工具            |
| -------- | --------------- |
| 代码开发 | GitLab / GitHub |
| CI/CD    | Jenkins         |
| 数据开发 | DataLeap        |

也就是说它提供：

### 1 数据集成（ETL）

把数据从各种地方拉过来：

```
MySQL
Kafka
日志
ClickHouse
Hive
```

做：

```
采集
同步
清洗
转换
```

典型工具类似：

- Airflow
- DataX
- Flink
- Spark

---

### 2 数据开发

数据工程师写 SQL 或任务：

例如：

```
每天统计用户留存
```

平台负责：

```
调度
运行
失败重试
资源分配
```

类似：

```
Airflow / DolphinScheduler
```

---

### 3 数据治理

解决数据混乱问题：

比如：

```
user_id 是什么
哪个表权威
数据是否过期
```

需要：

```
元数据管理
血缘关系
数据质量
```

---

### 4 数据资产管理

本质就是：

```
数据目录
数据搜索
权限管理
```

类似：

```
企业内部的数据 Google
```

---

总结一句：

> **DataLeap = 字节内部的数据开发操作系统**

所以这个岗位其实叫：

# 二、为什么 JD 里出现 Scala

你敏锐地注意到了一个关键点。

JD 写：

```
Java / Scala
Flink / Spark
```

这其实暴露了技术栈。

### 大数据生态

大部分核心系统是：

| 系统  | 语言         |
| ----- | ------------ |
| Spark | Scala        |
| Flink | Java / Scala |
| Kafka | Scala        |
| Akka  | Scala        |

原因很简单：

**Scala 是 JVM 上最适合写分布式框架的语言。**

它有：

```
函数式编程
类型系统
模式匹配
高并发模型
```

所以：

- **写框架的人用 Scala**
- **写业务的人用 Java**

DataLeap 这种平台，很可能：

```
底层引擎
→ Scala
服务层
→ Java
```

# 一、DataLeap 这种平台的真实架构

先看 **大数据平台的标准结构**。

```
数据源
  ↓
数据采集
  ↓
消息队列
  ↓
实时 / 离线计算
  ↓
数据仓库
  ↓
数据服务 / BI
```

换成系统组件：

| 层       | 典型组件                   |
| -------- | -------------------------- |
| 数据采集 | DataX / CDC                |
| 消息队列 | Kafka                      |
| 实时计算 | Flink                      |
| 离线计算 | Spark                      |
| 存储     | Hive / HDFS / ClickHouse   |
| 查询     | Presto / Trino             |
| 调度     | Airflow / DolphinScheduler |
| 元数据   | Data Catalog               |

DataLeap 本质就是：

> **把这些东西整合成一个平台。**

---

目标：

```

```

CDC = Change Data Capture

例如：

```

↓
Kafka
```

典型工具：

```

```


这几个概念必须会。

# 二、数据同步系统设计（面试高频）

面试非常喜欢问：

> **设计一个 MySQL → Kafka 的同步系统**

核心流程：

```
MySQL
  ↓
binlog
  ↓
解析
  ↓
Kafka
```

---

## 1 binlog解析

MySQL会记录：

```
insert
update
delete
```

binlog事件。

解析步骤：

```
读取binlog
↓
解析事件
↓
转换为JSON
↓
发送Kafka
```

---

## 2 数据一致性问题

必须处理：

### 重复消费

Kafka可能重复。

解决：

```
幂等写入
```

---

### 顺序问题

MySQL：

```
update1
update2
```

Kafka必须保证：

```
顺序一致
```

解决：

```
同一主键进入同一partition
```

---

### 断点恢复

同步系统崩溃。

需要：

```
保存binlog position
```

恢复时：

```
继续读取
```

---

## 3 扩展问题

面试可能继续问：

### 全量同步 + 增量同步

第一次同步：

```
全量数据
```

之后：

```
binlog增量
```
