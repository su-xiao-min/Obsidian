项目问题非常大，
我做完之后感觉很糟糕，
我就来说一下我认为存在的问题。

首先是技术栈的选择问题，
就是我前面提到的 DeepSeek OCR。
这个东西我其实不需要去部署它，
在我的 Java 项目中，
最好的选择是直接去调用一个 API 服务，
非常简单，
有很好的接口文档和调用指南。
但我恰恰选择了最愚蠢的方式，
还需要自己前后对接接口，
反复测试，
在这件事上花费了很多时间，
可它对项目的帮助其实并不大。
「从后来的结果来看，
没有一个面试官在意我怎样部署它的」

其他的还有搜索引擎的选择，
其实这个项目本来可以直接从 MySQL 数据库当中进行检索，
但我想不开，
选择了 Elasticsearch。
因为我读了一些书，
觉得这个搜索引擎性能很好，
是很高级的检索方式。
还是那个问题，
我的项目需要这么高的复杂性吗？
如果不需要，
加在这里除了炫技，
其实没什么用处，
而且我连炫技都没炫出来。
因为我能力的问题，
包括生成向量代码和最后把数据存入 Elasticsearch 时，
都出现了很多问题。
好在我是使用 Docker 部署的，
部署起来倒还没那么痛苦。

第三方面的问题是接口文档的问题。
因为我的项目涉及很多接口，
前端和后端的接口、
后端跟中间件的接口，
乃至一些外部服务是我自己部署的，
也需要接口。
但因为我自己没好好检查接口文档，
首先，
我调用阿里云嵌入的模型时，
弄错了嵌入的维度，
调用的模型返回的是 1024 维的数据，
但我在代码中一直按 2048 维来处理，
所以这一块一直跑不通，
查了半天才发现问题。
还有阿里云 OSS 服务的接口，
我没看清楚其官方接口文档就开始写代码，
按照普通的非对称签名的方式来写，
而且都没检查权限。
当我想要使用域签名基础时，
应该解决跨域问题，
但我没让 OSS 服务在我的 bucket 中，
所以当时半天调不通，
反复浪费时间。

最痛苦的当属部署 DeepSeek-OCR 服务的时候。
一开始我选的是直接在 DeepSeek-OCR 这里实现一个异步机制，
采用轮询，
之后才改成回调。
但改成回调后，
一些接口又出现混乱。
我在后端给 DeepSeek-OCR 服务传递接口文档时，
错误地设计了接口，
导致它传递给我的接口和我自己代码中的接口不一样，
但我一直没注意到这个问题，
一直在改代码，
新问题不断出现，
代码变得非常冗余。
其实仅仅是一点接口的问题，
如果一开始写接口文档时就想清楚需要怎样的接口，
就不用浪费这么多时间。

第四点就是异步机制的问题，
调用外部接口时，
如果外部接口返回速度很慢，
应该怎样解决？
实际上有很多好的解决方案，
比如使用 Netty，
或者使用回调，
都比我一开始选择的轮询要强很多。
但我没想清楚这个问题就开始写代码，
使用了愚蠢的轮询机制。
中途想改代码时，
因为自己部署了 DeepSeek-OCR 服务，
代码也需要改动。
但我改 DeepSeek-OCR 服务时也不用心，
使用了错误的提示词，
所以反馈的结果也有问题。
而且我在解析 DeepSeek-OCR 传递的接口时也有问题，
没进行格式校验，
没在日志中详细报错，
所以迟迟没注意到解析时会有问题。

另外，
我的前端代码展示过程中也存在问题，
这还是接口文档不清楚导致的。
当需要进行前后端联调开发，
乃至自己部署多端服务时，
接口文档真的非常重要，
技术选型也很重要。
我应该再多捋一遍，
把所有基础流程捋清楚，
再着手写代码、
进行项目设计，
而不是不断尝试、
写代码、
运行，
发现新问题再解决，
然后创造出更多问题。

确实，
要先把简单的问题解决，
让项目跑通，
但在项目跑通之前，
应该先把 Todo 捋清楚。
如果一开始就能把整个项目的构架想清楚，
后面可以省下很多时间。
更重要的是，
进行项目测试时，
不应该每次都整体编译、
运行项目，
这样效率太慢，
应该有更高效的项目测试方案，
但这些我都没弄好，
所以项目最后只实现了很简单的功能，
甚至没有实现一些权限机制的校验，
也没能实现更复杂的表的设计。
我感觉自己就像一个小丑。
