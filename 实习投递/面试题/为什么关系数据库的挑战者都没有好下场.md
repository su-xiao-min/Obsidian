MapReduce，是一种分布式的数据处理方式。SQL的放大版。因为Google的数据实在是太多了。

Apache Hadoop，当时很多，但是高射炮打蚊子，写起来很复杂。Hive顶层还是回到了SQL，但是低层的问题还是太多了。

Hadoop这种数据传递，会慢很多。系统灵活度总会在某个时刻超过性能。

GDB 缺乏标准的查询语言。17年才出现了标准的查询语言，很庆幸早早逃离了这个大坑。

图数据库和图算法混为一谈了。我们需要的其实是各种各样的图算法，我们需要的可能不是图数据库。

Wide Column Store CDB, 
宽表存储，
可以想象，一个非常空非常宽的表，轻松增加column，不占用空间。

意义很模糊。

Document Database MongoDB，很糟糕。原子能老师非常反感这个数据库。

它甚至连ACID都做不到，但是它一度很火。它把RDB的各种优点都贬低成了缺点，性能不佳。

这一层忽悠，吸引了很多人。RDB最懂规则表达了。

我们需要的不是一个schema-less的数据库，而不是一个可以储存不规则数据的RDB。走MongoDB的路，让MongoDB无路可走。

Vector DataBase，但是VDB，适应AI的出现来处理。

既然是这样，那么专门来管理的LLM的VDB的就是一切的重点了？

但是，我们搜索，不仅要考虑相似度，还需要考虑业务的逻辑，这一切都需要RDB。

在真正的项目当中，高度定制的业务逻辑，需要各种逻辑。所以，RDB厂商加入了对于vector，伟大的RDB。

伟大的RDB，我越来越喜欢数据库了。
