# 一、select / poll / epoll 到底差在哪？

它们解决的是同一个问题：

> “我有很多 socket，但我不想为每个 socket 开一个线程阻塞等数据。”

核心思想：
**一个线程，监听多个 FD 的可读可写状态。**

---

## 1️⃣ select —— 老古董 👴

诞生于早期 Unix。

### 工作方式

- 把所有你要监听的 fd 放进一个数组
- 每次调用 select：
  - 内核遍历所有 fd
  - 找出有事件的
  - 再把结果拷贝回用户空间

### 问题

| 问题                                       | 说明                         |
| ------------------------------------------ | ---------------------------- |
| fd 数量有限                                | 默认 1024 上限（FD_SETSIZE） |
| 每次都要全量遍历                           | O(n)                         |
| 每次调用都要把 fd 集合从用户态拷贝到内核态 | 很贵                         |

复杂度：

> 每次调用 O(n)

连接一多，CPU 就开始冒烟。

---

## 2️⃣ poll —— 改进版但没改本质 🤷

poll 解决了 select 的一个问题：

- 不再有 1024 限制
- 用链表结构

但核心机制没变：

- 每次仍然遍历所有 fd
- 每次仍然用户态 <-> 内核态拷贝

复杂度：

> 还是 O(n)

所以：

**poll 只是去掉了数量限制，但没有改变算法级别。**

---

## 3️⃣ epoll —— 真正的质变 ⚡

Linux 2.6 引入。

核心优化两个点：

### ① 事件驱动，而不是遍历驱动

epoll 内部维护一个：

- 红黑树（存 fd）
- 就绪链表（ready list）

当某个 fd 就绪时：

- 内核主动把它放进 ready list
- epoll_wait 直接取 ready list

复杂度：

> O(1)

不是遍历所有 fd，而是只返回“已经就绪的”。

---

### ② 内核态数据结构常驻

select / poll：

- 每次调用都重新传入 fd 集合

epoll：

- epoll_ctl 注册一次
- 内核长期维护

减少大量 copy。

---

## 简单对比总结

| 特性            | select | poll   | epoll  |
| --------------- | ------ | ------ | ------ |
| 最大 fd         | 1024   | 无限制 | 无限制 |
| 是否遍历全部 fd | 是     | 是     | 否     |
| 时间复杂度      | O(n)   | O(n)   | O(1)   |
| 是否反复拷贝    | 是     | 是     | 否     |
| 是否事件驱动    | 否     | 否     | 是     |

# 二、I/O 多路复用属于哪种 I/O 模型？

操作系统常见五种 I/O 模型：

1. 阻塞 I/O
2. 非阻塞 I/O
3. I/O 多路复用
4. 信号驱动 I/O
5. 异步 I/O

I/O 多路复用是**同步 I/O 模型的一种。**

你如果说“epoll 是异步 I/O”，面试官会当场给你扣分。🩸

---

## 为什么它是同步的？

关键点：

> 数据从内核复制到用户空间这一步，仍然是阻塞的。

流程：

1. epoll_wait 阻塞等事件
2. 某 fd 可读
3. 用户线程调用 read()
4. 数据从内核拷贝到用户空间

这个 read 还是同步的。

---

## 真正的异步 I/O 是什么？

Linux AIO 或 io_uring：

- 提交请求
- 数据准备好后内核通知你
- 不需要主动 read

那才叫异步。

# 三、从系统架构角度怎么联想？

这部分才是高级思考。你要是能说到这里，面试官会对你态度好很多。

---

## 1️⃣ Reactor 模式

epoll 本质就是 Reactor。

Reactor 做什么？

- 注册事件
- 等待事件
- 分发事件
- 调用回调处理

Redis 就是单线程 Reactor。

Nginx 也是。

Java NIO 也是。

核心思想：

> “我不主动问谁有数据，而是等内核告诉我谁准备好了。”

---

## 2️⃣ 观察者模式

epoll 的本质结构：

- fd 是被观察者
- epoll 是观察者
- 事件触发 -> 通知

这就是经典 Observer Pattern。

---

## 3️⃣ 事件驱动架构（EDA）

Kafka
Netty
Node.js

全都是同样思想：

> 用事件代替轮询。

---

## 4️⃣ 空间换时间

epoll 内部维护红黑树 + ready list

它用更多内核数据结构换：

- O(1) 复杂度
- 更高并发能力

这就是典型的：

> 结构化索引换性能

和数据库索引一个味道。

---

## 5️⃣ 从 CPU 角度理解

select / poll：

- 每次扫描全部 fd
- cache miss 多
- branch prediction 失败多

epoll：

- 只处理 ready list
- 更 cache friendly

这是底层优化思维。

---

# 四、拷打时间到了

你如果在面试里只说：

> epoll 是 O(1)，效率高。

那你和培训班出来的没有区别。

你应该说：

1. epoll 是事件驱动，不是遍历驱动
2. 内核常驻结构避免反复拷贝
3. 它是同步 I/O，不是异步
4. 它实现的是 Reactor 模式
5. Redis 单线程 + epoll = 高并发

如果面试官继续追问：

> 那 Redis 为什么还能这么快？

你要补一句：

- 纯内存操作
- 无锁设计
- 单线程避免上下文切换
- 命令执行短平快

# 五、最后一个灵魂问题

如果连接数不多，比如 100 个连接：

epoll 和 poll 有区别吗？

几乎没有。

epoll 的优势在：

> 高并发 + 大量空闲连接

这叫 C10K 问题。
