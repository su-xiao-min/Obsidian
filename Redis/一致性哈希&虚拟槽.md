# 一、先把几个概念彻底分清

你现在混在一起的是三种东西：

1. 取模分区（mod N）
2. 一致性哈希（Consistent Hash）
3. Redis Cluster 的虚拟槽（Hash Slot）

它们**不是一个东西**，只是都解决“key 怎么映射到节点”的问题。

---

# 二、最原始：取模分区

### 规则

```text
nodeIndex = hash(key) % N
```

这就是最简单的 HashMap 思维。

### 问题在哪里？

当 N 变化时：

* N = 4 时：key % 4
* N = 5 时：key % 5

**几乎所有 key 的结果都会变。**

👉 等价于：

> 扩容一次，全量数据重分布。

这在分布式数据库里是灾难。

你说“扩容必须 2 的幂”。那是 Java HashMap 为了优化位运算，但在分布式场景中不成立。
即便 2 的幂，节点从 4 变 8，还是要搬迁 50% 数据。

所以这种方案几乎不会用于真正的分布式数据库。

---

# 三、一致性哈希（Consistent Hash）

一致性哈希来自论文：

📄 Dynamo

思想：

* 把 hash 空间变成一个“环”
* key 和节点都映射到环上
* key 顺时针找到第一个节点

### 增加节点时

只会影响：

> 新节点和它前一个节点之间那一段数据

也就是说：

* 不再是全量重分布
* 只迁移一小段区间

这才是真正解决“扩容代价”的方案。

---

## 虚拟节点（Virtual Node）

一致性哈希通常会加：

> 一个物理节点映射多个虚拟节点

目的是：

* 解决数据倾斜
* 提高负载均衡

---

# 四、Redis Cluster 的虚拟槽是什么？

Redis Cluster **没有使用传统一致性哈希环。**

它用的是：

```text
固定 16384 个 hash slot
```

公式：

```text
slot = CRC16(key) % 16384
```

然后：

> 把 slot 分配给不同节点

这叫 **槽分片（Hash Slot Sharding）**

---

# 五、虚拟槽 vs 一致性哈希

你觉得它们一样，是因为：

它们都解决“扩容时少迁移”。

但实现方式不同。

| 对比     | 一致性哈希   | Redis 虚拟槽    |
| ------ | ------- | ------------ |
| 结构     | 环       | 固定槽数组        |
| 单位     | 虚拟节点    | 16384 slot   |
| 扩容方式   | 新节点抢一段环 | 新节点接管部分 slot |
| 数据迁移控制 | 自动      | 人工可控         |

Redis 的方案其实更简单。

本质是：

> 把数据映射问题拆成两层：
>
> key → slot
> slot → node

而一致性哈希是：

> key → node（一步到位）

---

# 六、Redis 在增删节点时具体做什么？

这才是你真正想问的。

---

## 1️⃣ 新增节点

假设原来 3 个节点。

新增一个节点时：

### 第一步：分配 slot

例如：

* 原节点 A：0–5000
* B：5001–10000
* C：10001–16383

现在加 D。

可能变成：

* A：0–4000
* B：4001–8000
* C：8001–12000
* D：12001–16383

👉 本质：

把一部分 slot 从旧节点迁移到新节点。

---

### 第二步：迁移数据

对于每一个要迁移的 slot：

1. 旧节点标记该 slot 为 MIGRATING
2. 新节点标记为 IMPORTING
3. 使用 `MIGRATE` 命令搬 key

迁移过程中：

* 客户端可能会收到 `ASK` 重定向
* 最终 cluster state 更新

这是**渐进式迁移**，不是停机重分布。

---

## 2️⃣ 删除节点

删除一个节点时：

* 把它的 slot 重新分配给其他节点
* 执行同样的 slot 迁移流程

逻辑是对称的。

---

# 七、为什么 Redis 不用一致性哈希？

几个原因：

### 1️⃣ 可控性

一致性哈希：

* 分布是“自动”的
* 不好精细控制

Redis slot：

* 运维可以精确指定哪个节点负责哪些 slot

---

### 2️⃣ 实现复杂度

一致性哈希需要维护环结构和虚拟节点。

Redis 只需要：

```text
16384 长度数组
```

简单、直接、稳定。

---

### 3️⃣ 客户端支持更好

Redis Cluster 客户端：

* 会缓存 slot → node 映射
* 直接定位目标节点
* 避免多跳

而传统一致性哈希通常在客户端维护环。

---

# 八、总结一句话

一致性哈希：

> 通过“环结构”减少扩容迁移。

Redis 虚拟槽：

> 通过“固定中间层 slot”实现可控迁移。

本质是：

> 一种是数学结构驱动
> 一种是工程抽象驱动

Redis 的做法更工程化。

---

Redis Cluster 的设计哲学接近：

* 两级映射

* 局部迁移
* 渐进式重平衡

它更像是一个“轻量级分布式系统”，而不是论文里那种学术模型。
