# Fenix BookStore 项目参数校验实现详解

## 项目概述

**项目名称**: Fenix BookStore Monolithic (单体书店系统)
**技术栈**:
- Spring Boot 2.2.5.RELEASE
- Java 1.8
- Bean Validation 2.0 (JSR 380)
- Hibernate Validator 6
- JAX-RS 2.1 (Jersey 2)
- Spring Data JPA

**文档目的**: 详细介绍本项目中的参数校验机制，包括标准校验、自定义校验注解、校验器实现以及异常处理机制。

## 一、校验框架基础

### 1.1 Bean Validation 2.0 (JSR 380)

本项目遵循 **JSR 380 Bean Validation 2.0** 标准，使用 **Hibernate Validator 6** 作为实现。Spring Boot Starter Web 已自动包含此依赖，无需在 `pom.xml` 中显式声明。

**核心特性**:
- 声明式校验（使用注解）
- 支持标准校验注解
- 支持自定义校验注解和校验器
- 可扩展的校验机制

### 1.2 校验触发机制

校验通过 `@Valid` 注解触发，支持：
- **方法级校验**: Resource层方法参数
- **字段级校验**: 实体类字段
- **级联校验**: 嵌套对象校验

## 二、标准Bean Validation注解使用

### 2.1 实体类字段校验

#### 2.1.1 Account 实体校验

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/domain/account/Account.java`

```java
@Entity
public class Account extends BaseEntity {
    @NotEmpty(message = "用户不允许为空")
    private String username;

    @NotEmpty(message = "用户姓名不允许为空")
    private String name;

    @Pattern(regexp = "1\\d{10}", message = "手机号格式不正确")
    private String telephone;

    @Email(message = "邮箱格式不正确")
    private String email;

    @Size(min = 6, max = 20, message = "密码长度必须在6-20位之间")
    private String password;
}
```

**校验规则说明**:
- `@NotEmpty`: 验证字符串不为 null 且长度大于 0
- `@Pattern`: 使用正则表达式验证手机号格式（中国11位手机号，以1开头）
- `@Email`: 验证邮箱格式
- `@Size`: 验证字符串长度在指定范围内

#### 2.1.2 Product 实体校验

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/domain/warehouse/Product.java`

```java
@Entity
public class Product extends BaseEntity {
```

- `@NotNull`: 验证值不为 null（适用于对象类型）
- `@Min`: 验证数值大于等于指定值

### 2.2 DTO (数据传输对象) 校验

## 三、自定义校验注解与校验器

### 3.1 自定义校验注解结构

自定义校验注解包含三个部分：
1. **注解定义**: 使用 `@Constraint` 指定校验器
2. **校验器实现**: 实现 `ConstraintValidator` 接口
3. **校验逻辑**: 在 `isValid()` 方法中实现具体的校验规则

### 3.2 账户相关自定义校验

#### 3.2.1 唯一账户校验 (@UniqueAccount)

**注解定义**: `src/main/java/com/github/fenixsoft/bookstore/domain/account/validation/UniqueAccount.java`

```java
@Documented
@Retention(RUNTIME)
@Target({FIELD, METHOD, PARAMETER, TYPE})
@Constraint(validatedBy = AccountValidation.UniqueAccountValidator.class)
public @interface UniqueAccount {
    String message() default "用户名称、邮箱、手机号码均不允许与现存用户重复";

    Class<?>[] groups() default {};

    Class<? extends Payload>[] payload() default {};
}
```

**使用场景**: 创建新用户时，确保用户名、邮箱、手机号与现有用户不重复。

#### 3.2.2 已认证账户校验 (@AuthenticatedAccount)


**使用场景**: 更新用户信息时，验证操作的用户是当前登录用户本人。

#### 3.2.3 无冲突账户校验 (@NotConflictAccount)

```java

@Target({FIELD, METHOD, PARAMETER, TYPE})
@Constraint(validatedBy = AccountValidation.NotConflictAccountValidator.class)
public @interface NotConflictAccount {
    String message() default "用户名称、邮箱、手机号码与现存用户产生重复";
```

**使用场景**: 更新用户信息时，允许用户修改自己的信息为自己已有信息，但不能与其他用户重复。

### 3.3 账户校验器实现 (AccountValidation)

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/domain/account/validation/AccountValidation.java`

```java
public class AccountValidation<T extends Annotation> implements ConstraintValidator<T, Account> {

    @Inject
    protected AccountRepository repository;

    protected Predicate<Account> predicate = c -> true;

    @Override
    public boolean isValid(Account value, ConstraintValidatorContext context) {
        // 避免在JPA持久化时重复验证
        // 当repository为null时，表示是在Hibernate持久化过程中，跳过验证
        return repository == null || predicate.test(value);
    }

    // 用户存在性校验
    public static class ExistsAccountValidator extends AccountValidation<ExistsAccount> {
        public void initialize(ExistsAccount constraintAnnotation) {
            predicate = c -> repository.existsById(c.getId());
        }
    }

    // 认证账户校验
    public static class AuthenticatedAccountValidator extends AccountValidation<AuthenticatedAccount> {
        public void initialize(AuthenticatedAccount constraintAnnotation) {
            predicate = c -> {
                Object principal = SecurityContextHolder.getContext().getAuthentication().getPrincipal();
                if ("anonymousUser".equals(principal)) {
                    return false;
                } else {
                    AuthenticAccount loginUser = (AuthenticAccount) principal;
                    return c.getId().equals(loginUser.getId());
                }
            };
        }
    }

    // 唯一账户校验
    public static class UniqueAccountValidator extends AccountValidation<UniqueAccount> {
        public void initialize(UniqueAccount constraintAnnotation) {
            predicate = c -> !repository.existsByUsernameOrEmailOrTelephone(
                c.getUsername(), c.getEmail(), c.getTelephone());
        }
    }

    // 无冲突账户校验
    public static class NotConflictAccountValidator extends AccountValidation<NotConflictAccount> {
        public void initialize(NotConflictAccount constraintAnnotation) {
            predicate = c -> {
                Collection<Account> collection = repository.findByUsernameOrEmailOrTelephone(
                    c.getUsername(), c.getEmail(), c.getTelephone());
                // 将用户名、邮件、电话改成与现有完全不重复的，或者只与自己重复的，就不算冲突
                return collection.isEmpty() ||
                    (collection.size() == 1 && collection.iterator().next().getId().equals(c.getId()));
            };
        }
    }
}
```

**设计亮点**:
1. **模板方法模式**: 使用基类 `AccountValidation` 封装通用逻辑，通过 `predicate` 函数式接口实现不同的校验策略
2. **避免重复验证**: 通过判断 `repository` 是否为 `null` 来避免在 Hibernate 持久化时重复验证
3. **Spring 依赖注入**: 使用 `@Inject` 注入 `AccountRepository`，访问数据库进行业务规则验证

### 3.4 库存充足性校验

#### 3.4.1 自定义注解 (@SufficientStock)

**注解定义**: `src/main/java/com/github/fenixsoft/bookstore/domain/payment/validation/SufficientStock.java`

```java
@Documented
@Retention(RUNTIME)
@Target({FIELD, METHOD, PARAMETER, TYPE})
@Constraint(validatedBy = SettlementValidator.class)
public @interface SufficientStock {
    String message() default "商品库存不足";

    Class<?>[] groups() default {};

    Class<? extends Payload>[] payload() default {};
}
```

#### 3.4.2 校验器实现

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/domain/payment/validation/SettlementValidator.java`

```java
public class SettlementValidator implements ConstraintValidator<SufficientStock, Settlement> {

    @Inject
    private StockpileService service;

    @Override
    public boolean isValid(Settlement value, ConstraintValidatorContext context) {
        return value.getItems().stream()
            .noneMatch(i -> service.getByProductId(i.getProductId()).getAmount() < i.getAmount());
    }
}
```

**校验逻辑**: 检查结算单中每个商品的购买数量是否小于库存数量。

## 四、Resource层校验触发

### 4.1 AccountResource 校验示例

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/resource/AccountResource.java`

```java
@Path("/accounts")
@Component
@CacheConfig(cacheNames = "resource.account")
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
public class AccountResource {

    @Inject
    private AccountApplicationService service;

    /**
     * 创建新的用户
     * @Valid 触发 Account 实体中定义的标准校验注解
     * @UniqueAccount 触发自定义唯一性校验
     */
    @POST
    @CacheEvict(key = "#user.username")
    public Response createUser(@Valid @UniqueAccount Account user) {
        return CommonResponse.op(() -> service.createAccount(user));
    }

    /**
     * 更新用户信息
     * @Valid 触发标准校验
     * @AuthenticatedAccount 验证当前登录用户
     * @NotConflictAccount 验证用户信息不与其他用户冲突
     */
    @PUT
    @CacheEvict(key = "#user.username")
    public Response updateUser(@Valid @AuthenticatedAccount @NotConflictAccount Account user) {
        return CommonResponse.op(() -> service.updateAccount(user));
    }
}
```

**关键点**:
- 使用 JAX-RS 标准注解 (`@Path`, `@POST`, `@PUT`)
- 通过 `@Valid` 触发 Bean Validation
- 支持同时使用多个自定义校验注解
- 校验失败时会抛出 `ConstraintViolationException`

## 五、异常处理机制

### 5.1 校验异常映射器

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/infrastructure/jaxrs/ViolationExceptionMapper.java`

```java
@Provider
public class ViolationExceptionMapper implements ExceptionMapper<ConstraintViolationException> {

    private static final Logger log = LoggerFactory.getLogger(ViolationExceptionMapper.class);

    @Override
    public Response toResponse(ConstraintViolationException exception) {
        log.warn("客户端传入了校验结果为非法的数据", exception);
        String msg = exception.getConstraintViolations().stream()
            .map(ConstraintViolation::getMessage)
            .collect(Collectors.joining("；"));
        return CommonResponse.send(Response.Status.BAD_REQUEST, msg);
    }
}
```

**功能说明**:
- 实现 JAX-RS 的 `ExceptionMapper` 接口
- 捕获 `ConstraintViolationException` 异常
- 提取所有校验失败的消息，用分号连接
- 返回 HTTP 400 Bad Request 状态码
- 记录警告日志，便于问题追踪

### 5.3 统一响应格式

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/infrastructure/jaxrs/CommonResponse.java`

```java
public class CommonResponse {
    public static Response send(Response.Status status, String message) {
        Map<String, Object> result = new HashMap<>();
        result.put("timestamp", System.currentTimeMillis());
        result.put("status", status.getStatusCode());
        result.put("message", message);
        return Response.status(status).entity(result).build();
    }

    public static Response failure(String message) {
        return send(Response.Status.INTERNAL_SERVER_ERROR, message);
    }

    public static Response op(Supplier<?> operation) {
        try {
            Object result = operation.get();
            return send(Response.Status.OK, "操作成功");
        } catch (Exception e) {
            return failure(e.getMessage());
        }
    }
}
```

**响应示例**:
```json
{
  "timestamp": 1738543200000,
  "status": 400,
  "message": "结算单中缺少商品清单；配送信息中缺少支付方式"
}
```

## 六、Service层的设计原则

### 6.1 Application Service 层职责

**文件位置**: `src/main/java/com/github/fenixsoft/bookstore/applicaiton/AccountApplicationService.java`

```java
@Named
@Transactional
public class AccountApplicationService {

    @Inject
    private AccountRepository repository;

    @Inject
    private Encryption encoder;

    public void createAccount(Account account) {
        // 不再进行参数校验，所有校验已在Resource层完成
        account.setPassword(encoder.encode(account.getPassword()));
        repository.save(account);
    }

    public Account findAccountByUsername(String username) {
        return repository.findByUsername(username);
    }

    public void updateAccount(Account account) {
        // 直接信任传入的参数，不重复校验
        repository.save(account);
    }
}
```

**设计原则**:
1. **单一职责**: Service层专注于业务逻辑协调，不处理校验
2. **信任输入**: 假设传入的参数已经过Resource层校验
3. **事务边界**: 使用 `@Transactional` 确保数据一致性
4. **避免重复**: 不在Service层重复Resource层的校验逻辑

## 七、避免重复校验的技巧

### 7.1 问题背景

当使用 JPA + Bean Validation 时，可能会遇到**重复校验**问题：
1. **第一次校验**: Resource层通过 `@Valid` 触发
2. **第二次校验**: JPA 持久化时，Hibernate 自动触发校验

这会导致自定义校验器被执行两次，可能引发性能问题或错误。

### 7.2 解决方案

**利用Spring和Hibernate的初始化时机差异**:

```java
public class AccountValidation<T extends Annotation> implements ConstraintValidator<T, Account> {

    @Inject
    protected AccountRepository repository;

    @Override
    public boolean isValid(Account value, ConstraintValidatorContext context) {
        // repository == null 时，表示是在Hibernate持久化过程中
        // 此时跳过验证，避免重复校验
        return repository == null || predicate.test(value);
    }
}
```

**原理**:
- **Spring容器**创建校验器时，会注入 `repository`（不为null）
- **Hibernate**持久化时创建的校验器，不会经过Spring容器，`repository` 保持为 `null`
- 通过判断 `repository` 是否为 `null`，区分校验触发场景

## 八、最佳实践总结

### 8.1 校验注解使用指南

| 场景 | 推荐注解 | 示例 |
|------|---------|------|
| 非空字符串 | `@NotEmpty` | `@NotEmpty String username` |
| 非空对象 | `@NotNull` | `@NotNull Integer amount` |
| 格式验证 | `@Pattern`, `@Email` | `@Pattern(regexp="1\\d{10}")` |
| 数值范围 | `@Min`, `@Max` | `@Min(0) Double price` |
| 长度限制 | `@Size` | `@Size(min=6, max=20) String password` |
| 唯一性约束 | 自定义注解 | `@UniqueAccount Account user` |
| 业务规则 | 自定义注解 | `@SufficientStock Settlement settlement` |

### 8.2 自定义校验注解开发步骤

1. **定义注解**:
   ```java
   @Constraint(validatedBy = YourValidator.class)
   public @interface YourCustomValidation {
       String message() default "默认错误消息";
       Class<?>[] groups() default {};
       Class<? extends Payload>[] payload() default {};
   }
   ```

2. **实现校验器**:
   ```java
   public class YourValidator implements ConstraintValidator<YourCustomValidation, YourType> {
       @Override
       public boolean isValid(YourType value, ConstraintValidatorContext context) {
           // 实现校验逻辑
           return true/false;
       }
   }
   ```

3. **使用注解**:
   ```java
   @Valid @YourCustomValidation YourType object
   ```

### 8.3 异常处理建议

1. **区分日志级别**:
   - `ConstraintViolationException`: 使用 `log.warn()`（客户端输入错误）
   - 其他异常: 使用 `log.error()`（系统内部错误）

2. **友好错误消息**:
   - 在注解的 `message` 属性中提供清晰的错误描述
   - 多个校验失败时，用分号连接所有消息

3. **统一响应格式**:
   - 包含时间戳、状态码、错误消息
   - 便于前端统一处理

## 九、常见问题与解决方案

### Q1: 为什么自定义校验器中的 repository 为 null？

**原因**: Hibernate 在持久化时也会触发校验，但此时创建的校验器不经过Spring容器。

**解决**: 参考第七节的方案，通过判断 `repository` 是否为 `null` 来跳过Hibernate持久化时的校验。

### Q2: 如何实现级联校验？

**解决**: 在嵌套对象字段上添加 `@Valid` 注解：
```java
public class Settlement {
    @Valid
    @NotNull
    private Purchase purchase;
}
```

### Q3: 如何根据不同场景使用不同的校验规则？

**解决**: 使用 **Validation Groups**：
```java
public interface CreateGroup {}
public interface UpdateGroup {}

public class Account {
    @Null(groups = CreateGroup.class)
    @NotNull(groups = UpdateGroup.class)
    private Integer id;
}

// 使用
@Validated(CreateGroup.class) Account account
```
